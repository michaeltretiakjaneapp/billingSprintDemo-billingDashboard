Claim Submission

How a biller sends a completed claim from Jane's Unsubmitted folder through the clearinghouse to the insurer — covering the pre-flight check, batch generation, EDI transmission, clearinghouse processing, and acknowledgement monitoring. Includes the Stedi pilot context and parallel Canadian portal submission path. Signal from #support_usa_billing, #support_usa_clearinghouses, #pt-insurance-foundations, #proj-stedi, and #pt-insurance.

G1 Biller can select and submit a batch of claims in as few clicks as possible without manual EDI file management
G2 Claims are transmitted to the right insurer and the biller gets immediate confirmation they were accepted — not a status mystery
G3 Clearinghouse-level rejections are surfaced in Jane with plain-language explanations, not raw EDI codes
0
Select Claims to Submit
Step 0.1
Biller opens Unsubmitted folder and selects claims for the batch
Biller
Task
Filter Unsubmitted folder by insurer, date, or patient and select the claims to batch
▶
Pain Point
The "View / Submit Selected" button label is confusing and non-intuitive — clinics regularly mistake it for something else, or choose the wrong export option (CMS-1500 paper form instead of EDI), sending claims through the wrong channel
Community"Have y'all experienced onboarding customers who find the 'View/Submit Selected' button a little confusing? I have had a fair amount of clinics not know how to proceed — or select Export > Batch CMS1500 instead of the EDI." — John Whaley, #support_usa_billing
Suggested Opportunity
Rename "View / Submit Selected" to something more explicit — "Generate EDI File" or "Submit Claims Electronically" — to reduce the confusion between EDI submission and paper form export that shows up repeatedly in onboarding.
Biller
Task
Review claim count and spot any claims with unsaved changes before submitting
▶
Pain Point
Claims with unsaved changes were previously submittable — causing an Internal Server Error mid-batch. Jane now blocks submission for claims with unsaved changes and shows a badge, but billers must still identify and fix them individually before the batch can proceed
Community"If a clinic has a purchase in the unsubmitted folder that has some unsaved changes, they won't be able to generate an EDI if that purchase is selected in their batch — the View/Submit Selected button will disappear." — John Whaley, #support_usa_billing
1
Pre-Flight Check
Step 1.1
Jane runs pre-flight validation before generating the EDI
Jane System
Task
Check that core required fields are present — NPIs, CPT codes, ICD-10 codes — and flag any missing data
▶
Pain Point
Jane's pre-flight check only catches basic omissions (missing NPI, missing codes). More nuanced issues — wrong modifier combinations, billing/rendering NPI collision, invalid ICD-10/CPT pairing — pass pre-flight and only surface as clearinghouse rejections after transmission
Community"Jane will do a basic pre-flight checklist to make sure some core pieces of information are not omitted... the clearinghouse will also do a check for more niche things that might cause a rejection — missing modifiers, codes that shouldn't be billed together." — Mosh, #support_usa_clearinghouses
Opportunity
Expanding pre-flight to include configurable Roux-style rules — modifier checks, NCCI edit warnings, NPI collision detection — would catch a broader set of avoidable rejections before the EDI leaves Jane.
Biller
Task
Set resubmission code and original reference number (for corrected claims only)
▶
Pain Point
Resubmission Code 6 ("Corrected Claim") is still selectable in the pre-flight but is outdated — selecting it causes the clearinghouse to reject the claim. The correct code is 7, but there is no guidance in the UI pointing billers to the right choice
Community"Resubmission Code 6 is an outdated feature! If they select this, the clearinghouse will reject the claim. Be sure to give that little extra pro-tip to help make sure they don't hit any roadblocks." — Karen, #support_usa_billing
Suggested Opportunity
Deprecate or grey out Resubmission Code 6 in the pre-flight UI, replacing it with a tooltip explaining why Code 7 is the correct choice for corrected claims today.
2
Generate EDI & Transmit to Clearinghouse
Step 2.1
Jane generates the 837 EDI file and sends it via SFTP to ClaimMD
Jane System
Task
Assign submission ID, generate 837 file, and transmit via SFTP to ClaimMD
▶
Opportunity
Jane assigns a unique submission ID to every claim at this step — this ID follows the claim all the way through and links back to insurer responses. The Stedi pilot moves toward a fully API-based integration that eliminates the SFTP handoff entirely, enabling real-time status updates rather than batch file polling.
Clearinghouse
Task
Receive EDI file, apply edits and validation, forward to insurer (or sub-clearinghouse)
▶
Pain Point
ClaimMD sometimes routes claims through a sub-clearinghouse (e.g. Availity) for certain payers without the clinic knowing — the claim appears to go directly to the insurer in Jane, but actually travels through an intermediary, which can cause unexpected acknowledgement delays or routing confusion
Community"The journey of the claim through the Acknowledgements shows how it started in Jane, then was transmitted to ClaimMD, who transmitted it to Availity, who forwarded it to the payer." — Mosh, #support_usa_billing
Suggested Opportunity
Surface the full clearinghouse routing path in the claim summary — so billers can see exactly where a claim is in its journey without guessing why a 277 acknowledgement is taking longer than expected.
Step 2.2
ClaimMD "Transmit Approval Required" gate (if enabled)
Biller
Task
Log into ClaimMD portal and approve claims for transmission (if Transmit Approval Required is on)
▶
Pain Point
Many new clinics unknowingly have "Transmit Approval Required" enabled in ClaimMD — claims appear to submit in Jane but sit silently in ClaimMD awaiting manual approval, sometimes for days or weeks before the clinic notices nothing went out
Community"They have 54 claims sitting in their ClaimMD account awaiting approval for transmit... If they want the claims to go through automatically, having this off would allow that." — Katie, #support_usa_billing
Opportunity
Jane could detect the "Transmit Approval Required" state from ClaimMD and surface a persistent warning in the clearinghouse settings — making it visible in Jane that claims are waiting in ClaimMD rather than appearing to have been submitted.
3
Monitor Acknowledgements
Step 3.1
Jane receives 277 acknowledgement from clearinghouse / insurer
Jane System
Task
Parse 277 response and update claim status in the Claim Summary panel
▶
Pain Point
Three distinct outcomes all look similar in the Unsubmitted / acknowledgement view: clearinghouse-level rejection (before insurer sees it), insurer-level rejection (after routing, before adjudication), and acceptance for adjudication. Billers must understand the lifecycle to know which type of action is needed for each
Community"This is the payer sending an update that the claim has been rejected and did not make it through to adjudication. The more specific rejection reasons are below." — Mosh, explaining 277 acknowledgement types, #support_usa_clearinghouses
Suggested Opportunity
Distinguish the three acknowledgement states visually in the Claim Summary panel — clearinghouse rejection (fix before insurer sees it), insurer rejection (fix and resubmit with correction code), and accepted for adjudication (wait for ERA) — with a plain-language action recommendation for each.
Biller
Task
Review acknowledgement status in the Claim Summary panel and determine next action
▶
Pain Point
Acknowledgement messages use raw EDI terminology ("ACKNOWLEDGEMENT RETURNED AS UNPROCESSABLE CLAIM — CLAIM ENCOUNTER HAS BEEN REJECTED AND HAS NOT BEEN ENTERED INTO THE ADJUDICATION SYSTEM") — completely opaque to non-technical billers who don't know what "adjudication system" means
Community"'PAYER RESPONSE: ACKNOWLEDGEMENT RECEIPT — THE CLAIM ENCOUNTER HAS BEEN RECEIVED. THIS DOES NOT MEAN THAT THE CLAIM HAS BEEN ACCEPTED FOR ADJUDICATION. ACCEPTED FOR PROCESSING.'" — Real acknowledgement text shown to clinics, #support_usa_clearinghouses
Opportunity
Plain-language translations of the most common 277 acknowledgement messages — surfaced inline in Jane as "What this means" and "What to do next" — would eliminate the most common category of clearinghouse support escalations.
Step 3.2
Biller confirms claim is accepted for adjudication — or identifies the rejection
Biller
Task
Confirm accepted claim — wait for ERA. Or identify rejection type and take corrective action
▶
Pain Point
Editing a claim after it's been submitted — even something as small as changing a diagnosis code order — automatically moves it back to Unsubmitted. If the biller doesn't notice and runs the next batch, the claim can get submitted twice, causing a duplicate rejection ("OA 18 — Exact duplicate claim")
Community"The diagnosis codes were modified. When they were, Jane moved the claim back to the Unsubmitted folder automatically. It was then batched together again and submitted a second time... the insurer received a duplicate, adjudicating one and rejecting the other." — Mosh, #support_usa_billing
Suggested Opportunity
When a submitted claim is edited and bounced back to Unsubmitted, surface a prominent "This claim was previously submitted on [date] — submitting again will send a duplicate. Did you mean to create a Corrected Claim?" warning to prevent accidental double submissions.
Biller
Task
Confirm the Claim Summary panel shows the full submission history accurately
▶
Pain Point
The Claim Summary panel has known gaps: no claim-level or policy-level notes field, and no clean distinction between denials (post-adjudication) and rejections (pre-adjudication) — the two most commonly cited gaps since the panel was first shipped
Community"There are some real gaps. The two that come up most are notes (at both the claim and policy level) and a cleaner way to handle denials vs rejections." — John Whaley, #pt-insurance
Opportunity
Adding claim-level notes to the Claim Summary panel — with a denial/rejection distinction — are the two highest-signal improvements to the post-submission tracking experience, according to John Whaley's own assessment of the gaps.
4
Alternate Path: Stedi Pilot & Future Integrated Submission
Step 4.1
Clinic submits claims via Stedi (pilot) — API-based, no SFTP handoff
Jane System
Task
Submit 837 directly via Stedi API and receive real-time 277CA acknowledgement
▶
Opportunity
The Stedi pilot (currently onboarding first clinic, Rebuilt Strength + Rehab) replaces the SFTP batch model with a direct API integration — enabling real-time acknowledgements, embedded claim tracking, and an eventual future where clinics never need to leave Jane to manage submission status.
Feedback"Our vision hasn't changed: a world where Jane customers never need to log into a separate clearinghouse account." — John Whaley, #proj-insurance-ai
Pain Point
Stedi is newer and has less payer-specific claim editing logic than legacy clearinghouses — some rejections that ClaimMD auto-corrects may surface as errors in Stedi that require manual biller action. The pilot is specifically designed to surface these gaps before broader rollout
Community"Legacy clearinghouses have years of logic tweaking EDI information to avoid rejections. Stedi's newer, so we're testing with about 10 select clinics starting late February. It's intentionally quiet." — John Whaley, #pt-insurance-leads
Step 4.2
Canadian path: biller submits claims via insurer portal (TELUS eClaims, SunLife, etc.)
Biller
Task
Log into insurer portal, manually re-enter patient and claim details, submit
▶
Pain Point
For Canadian insurers without a Jane integration, billers must log into each insurer's web portal separately and re-enter all claim details manually — patient name, DOB, policy ID, CPT equivalent codes — every time, with no data flowing from Jane
Community"Manual data entry is the biggest pain — name, DOB, policy ID, every single time. We think there is no saving and no pre-fill in the Sun Life or providerNet portal." — Maddie Maerz, #ai-sdk-hack-cad-portal-submission
Opportunity
The AI-SDK hackathon project (#ai-sdk-hack-cad-portal-submission) prototyped an agent-based portal submission flow using Stagehand — automating the full eClaims portal login → new claim → fill → submit flow from within Jane, eliminating manual re-entry entirely for Canadian insurer portals.
